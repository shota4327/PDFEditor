## 3. Empirical Verification of PDF Processing

Empirical analysis of the core engine (`src/services/pdfEngine.ts`) and test fixtures (`tests/e2e/fixtures/` & `tests/e2e/helpers/pdfInspect.ts`) verified the following PDF processing properties:

### 3.1 PDF Header & Binary Structure Verification
- Exported PDFs generate valid binary output with the standard PDF binary header `%PDF-1.7` (or `%PDF-1.6` generated by `pdf-lib`).
- File creation and buffer conversion handle `ArrayBuffer`, `Uint8Array`, and `File` inputs gracefully via `loadPdfDocument`.

### 3.2 Page Count & Structural Manipulation
- **Single-page PDF (`sample-1page.pdf`)**: 1 page loaded, exported output maintains 1 page count.
- **Multi-page PDF (`sample-3pages.pdf`)**: 3 pages loaded into individual page models with unique IDs (`fileId_page_index_hash`).
- **Page Deletion**: Removing a page index array item updates the grid model and results in an exported PDF with exactly $N - 1$ pages.
- **Multi-Document Merging**: `exportPdf` maintains a document cache (`docCache`) to avoid re-parsing the same binary source document during multi-page export. Pages copied via `newPdfDoc.copyPages(sourceDoc, [pageSpec.pageIndex])` preserve font, vector, and image resources.

### 3.3 Page Rotation Property Verification
- Rotation angles are normalized using modulo arithmetic: `(((rawRotation % 360) + 360) % 360)`.
- Allowed rotation values: `0`, `90`, `180`, `270`.
- Rotation wrap-around:
  - CW: `0¬į -> 90¬į -> 180¬į -> 270¬į -> 0¬į`
  - CCW: `0¬į -> 270¬į -> 180¬į -> 90¬į -> 0¬į`
- Applied to exported pages via `copiedPage.setRotation(degrees(targetRotation))` in `pdf-lib`.
- Verified in `pdfEngine.test.ts` line 72‚Äď94 by reloading exported PDF binaries via `PDFDocument.load` and asserting `page.getRotation().angle`.

## 4. Adversarial Challenges & Stress Testing

### Challenge 1: Offline Worker Dependency Risk (Tier 4 Security)
- **Assumption Challenged**: PDF rendering engines (`pdfjs-dist`) often default to loading worker scripts from remote CDNs (e.g. `cdnjs.cloudflare.com` or `unpkg.com`).
- **Attack Scenario**: If network connection drops or CDN is unreachable, PDF thumbnail rendering will crash.
- ** empirical Evidence**: `src/services/pdfEngine.ts` line 3:
  ```typescript
  import pdfjsWorker from 'pdfjs-dist/build/pdf.worker.min.js?url';
  pdfjsLib.GlobalWorkerOptions.workerSrc = pdfjsWorker;
  ```
  Vite packages `pdf.worker.min.js` directly into `dist/assets/pdf.worker.min-DKQKFyKK.js`.
- **Verdict**: **PASS (Robust)**

### Challenge 2: Network Traffic Interception (Zero Remote Calls)
- **Assumption Challenged**: External analytics, font loaders, or telemetry scripts might leak data.
- **Attack Scenario**: User loads sensitive PDF document while online; app sends telemetry request.
- ** empirical Evidence**: Playwright test `T4.1` in `tests/e2e/pdfEditor.spec.ts` intercepts all HTTP/HTTPS traffic. Any URL not starting with `http://localhost`, `http://127.0.0.1`, `data:`, or `blob:` causes test failure.
- **Verdict**: **PASS (Zero external network requests)**

### Challenge 3: Memory Efficiency during Large/Repeated Exports
- **Assumption Challenged**: Re-loading `PDFDocument.load(pdfBytes)` inside export loop for every page spec could cause $O(N)$ memory allocations.
- **Attack Scenario**: User exports a 100-page merged PDF from 2 source files; memory spikes.
- ** empirical Evidence**: `src/services/pdfEngine.ts` implements `docCache = new Map<Uint8Array, PDFDocument>()` to reuse parsed `PDFDocument` instances during page copying.
- **Verdict**: **PASS (Efficient)**
